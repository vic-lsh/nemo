# Nemo FPGA design

CXL-based tiered memory implemented on top of Intel's R-tile CXL Type-3
(memory expander, DDR4) reference example design.

This repository **only ships Nemo-authored content** — the Intel example
design and IP wrappers are not redistributed here. To build the design,
you must regenerate the Intel reference example yourself from Quartus
(see [Step 1](#1-regenerate-the-intel-example-design)) and then overlay
this tree on top of it.

## Requirements

- **Quartus Prime Pro 24.1.0** (Build 115, SC Pro Edition). Earlier or
  newer Quartus releases ship different versions of the R-tile CXL IP
  and the example design — they are not guaranteed to be patch-compatible.
- **Target device**: Agilex 7 `AGIB027R29A1E2VB`.
- **Questa-Intel FPGA Edition** (bundled under
  `<quartus>/questa_fse/`) with a valid Questa license for simulation.
  Set `LM_LICENSE_FILE` to point at it.
- The standard Quartus environment variables — sourcing the install's
  `adm/qenv.sh` is the simplest way to set `QUARTUS_ROOTDIR` and `PATH`.
- For host-side tooling: a Linux box with `libnuma-dev` (see
  `host/scripts/install_deps.sh`) and matching kernel headers for the
  `host/memmod/` kernel module.

## Bringing up the design

### 1. Regenerate the Intel example design

The R-tile CXL Type-3 IP and its example design are part of Quartus
Prime Pro. Refer to the *Intel FPGA R-tile CXL IP User Guide* for your
Quartus version for the authoritative procedure. The high-level steps:

1. **Generate the CXL IP** at `fpga/intel_rtile_cxl_top.ip`:
   - Open Quartus → IP Catalog → search for "R-tile Hard IP for CXL".
   - Configure as Type-3 (memory expander), DDR4 backing memory, target
     device `AGIB027R29A1E2VB` (Agilex 7).
   - Save the generated IP at `fpga/intel_rtile_cxl_top.ip`. This step
     also creates `fpga/intel_rtile_cxl_top/` containing the Verilog
     wrapper.
2. **Generate the matching example design**:
   - From the IP editor, click *Generate Example Design*.
   - Target output: `fpga/intel_rtile_cxl_top_0_ed/`.
   - This populates `hardware_test_design/`,
     `intel_rtile_cxl_top_cxltyp3_ed/`, `cxl_ed_tb/`, `sim_filelist/`,
     and the `cxltyp3_memexp_ddr4_top.qpf`/`.qsf` Quartus project that
     `scripts/build.sh` invokes.

After Step 1, the Nemo files already in `intel_rtile_cxl_top_0_ed/`
(`common/custom/nemo/` and `sim/sequences/`) sit alongside the freshly
regenerated Intel content. The original Intel example design **must not
overwrite** those two subdirectories — if Quartus prompts about
existing files, keep ours.

Nemo's baseline corresponds to the example design generated by Quartus
Prime Pro 24.1.0 (Build 115, around May 2024). If your Quartus release
emits a slightly different baseline, the patches in Step 2 may need a
manual touch-up.

### 2. Apply the Nemo patches

Five Intel-supplied files are modified by Nemo. Apply the patches from
`fpga/`:

```sh
cd fpga
for p in patches/*.patch; do
    patch -p1 < "$p"
done
```

| Patch | Target | Purpose |
| --- | --- | --- |
| `afu_top.sv.patch` | `intel_rtile_cxl_top_0_ed/hardware_test_design/common/afu/afu_top.sv` | Instantiate `nemo_top`, wire AXI/CSR, fix AXI-channel safety. |
| `cxltyp3_memexp_ddr4_top.qsf.patch` | `intel_rtile_cxl_top_0_ed/hardware_test_design/cxltyp3_memexp_ddr4_top.qsf` | Add Nemo sources to synthesis; multipipeline tweaks. |
| `afu_top_tb.sv.patch` | `intel_rtile_cxl_top_0_ed/hardware_test_design/sim/afu_top_tb.sv` | Extend Intel TB for pipelining / tracking mode 2. |
| `afu_top_tb_axilite_interface.sv.patch` | `intel_rtile_cxl_top_0_ed/hardware_test_design/sim/afu_top_tb_axilite_interface.sv` | TB AXI-lite harness extensions. |
| `axi_slave_example.sv.patch` | `intel_rtile_cxl_top_0_ed/hardware_test_design/sim/axi_slave_example.sv` | TB AXI slave helper extensions. |

### 3. Bump HDM size to 32 GB

The stock example design ships with a 16 GB HDM (host-defined memory)
size. Nemo targets 32 GB. Apply the following one-line edits to the
Intel-supplied sources you regenerated in Step 1. We can't ship these
as a patch (they would embed Intel source code), but each is a single
parameter change.

**`ed_top_wrapper_typ3.sv`** — under the `\`ifdef HDM_64G` / `\`else`
block (around line 1105), change the `\`else` branch:

```
- assign hdm_size_256mb = 36'h40; // HDM_16G
+ assign hdm_size_256mb = 36'h80; // HDM_32G
```

**`intel_rtile_cxl_top_cxltyp3_ed/intel_rtile_cxl_top_1120/sim/cxlip_top_pkg.sv`**
and **`…/synth/cxlip_top_pkg.sv`** — both have an identical block
around line 81; change:

```
- localparam EMIF_AMM_ADDR_WIDTH = 27;
+ localparam EMIF_AMM_ADDR_WIDTH = 28;
```

**`common/cafu_csr0/cafu_csr0_cfg.sv`** — three small DVSEC tweaks:

- `DVSEC_DEV.next_cap_offset`: `12'hF40` → `12'hF60`
- `DVSEC_FBRANGE1SZLOW.media_type`: `POR_DVSEC_FBRANGE1SZLOW_media_type` → `3'b000`
- `DVSEC_FBRANGE1SZLOW.memory_class`: `POR_DVSEC_FBRANGE1SZLOW_memory_class` → `3'b000`

### 4. Synthesize

```sh
cd fpga
./scripts/build.sh
```

This invokes `quartus_sh --flow compile` on
`intel_rtile_cxl_top_0_ed/hardware_test_design/cxltyp3_memexp_ddr4_top.qpf`
(the Intel-supplied Quartus project, now patched). Build log is teed to
`build.log` and timing is checked at the end. The bitstream lands in
`intel_rtile_cxl_top_0_ed/hardware_test_design/output_files/`.

For a syntax-only check (elaboration, no place-and-route), use
`./scripts/check_syntax.sh` instead — it's much faster.

### 5. Program the FPGA

```sh
cd fpga
./scripts/program_fpga.sh
```

Programs the device via `quartus_pgm` using the `.cdf` config in the
Intel output_files directory. Pinned to all but the last CPU core with
real-time priority to speed bitstream load.

### 6. Simulate

```sh
cd fpga
source <quartus>/adm/qenv.sh     # sets QUARTUS_ROOTDIR, etc.
export LM_LICENSE_FILE=<path-to-questa-license>
./run_sim.sh intel_rtile_cxl_top_0_ed/hardware_test_design/sim/sequences/<sequence>.py
```

`run_sim.sh` takes a Python testbench-sequence generator as its sole
argument. Available sequences live in
`intel_rtile_cxl_top_0_ed/hardware_test_design/sim/sequences/` (e.g.
`vanilla.py`, `cxl_controlpath.py`, `translation_unit.py`). The Python
script emits a generated `.sv` testbench, which `run_sim.do` then
elaborates against the Altera primitives (resolved via
`$QUARTUS_ROOTDIR` in `lib_filelist.txt`).

See `docs/sim.md` for waveform viewing (`view_sim.sh`, `view_wave.sh`)
and GUI sim notes.

## Host-side tooling

```sh
cd fpga/host
./scripts/install_deps.sh        # apt-get libnuma-dev
make                             # builds host/src/*.c into host/bin/
(cd memmod && make)              # builds memmod/memtest_module.ko
```

To configure the CXL memory exposed by the FPGA as a DAX device on the
host, follow `docs/cxl_dax_setup.md`. The `host/setup_cxl_as_dax.sh`
script automates the final `ndctl create-namespace` step.

## Continuous integration

CI runs on a UW-internal server (Quartus licenses are not portable to
GitHub-hosted runners). Push branches whose names start with `build/`
trigger full builds; every push triggers a syntax check via
`scripts/check_syntax.sh`. See `docs/ci.md` for the full flow.
